# Programming Specification — OOP & Collections

Verbatim excerpts from https://alibaba.github.io/Alibaba-Java-Coding-Guidelines/

---

## OOP Rules

**1. [Mandatory]** A static field or method should be directly referred to by its class name instead of its corresponding object name.

**2. [Mandatory]** An overridden method from an interface or abstract class must be marked with `@Override` annotation.

> Counter example: For `getObject()` and `get0bject()`, the first one has a letter 'O', and the second one has a number '0'. To accurately determine whether the overriding is successful, an `@Override` annotation is necessary.

**3. [Mandatory]** `varargs` is recommended only if all parameters are of the same type and semantics. Parameters with `Object` type should be avoided.

> Note: Arguments with the varargs feature must be at the end of the argument list.
>
> Positive example:
> ```java
> public User getUsers(String type, Integer... ids);
> ```

**4. [Mandatory]** Modifying the method signature is forbidden to avoid affecting the caller. A `@Deprecated` annotation with an explicit description of the new service is necessary when an interface is deprecated.

**5. [Mandatory]** Using a deprecated class or method is prohibited.

> Note: For example, `decode(String source, String encode)` should be used instead of the deprecated method `decode(String encodeStr)`.

**6. [Mandatory]** Since `NullPointerException` can possibly be thrown while calling the `equals` method of `Object`, `equals` should be invoked by a constant or an object that is definitely not `null`.

> Positive example: `"test".equals(object);`
>
> Counter example: `object.equals("test");`
>
> (`Objects.equals(a, b)` is also acceptable — added in JDK 7.)

**7. [Mandatory]** Use the `equals` method, rather than reference equality `==`, to compare primitive wrapper classes.

> Note: When the value fits the range from −128 to 127, we can use `==` directly. Because the `Integer` object will be generated by `IntegerCache.cache`. Outside that range, `==` will silently return `false` for equal values.

**8. [Mandatory]** To judge the equivalence of floating-point numbers, `==` cannot be used for primitive types, while `equals` cannot be used for wrapper classes.

> Counter example:
> ```java
> float a = 1.0f - 0.9f;
> float b = 0.9f - 0.8f;
>
> if (a == b) {
>     // seems like this block will be executed
>     // but the result of a == b is false
> }
>
> Float x = Float.valueOf(a);
> Float y = Float.valueOf(b);
> if (x.equals(y)) {
>     // seems like this block will be executed
>     // but the result of x.equals(y) is false
> }
> ```
>
> Positive example (1) — primitive with epsilon:
> ```java
> float a = 1.0f - 0.9f;
> float b = 0.9f - 0.8f;
> float diff = 1e-6f;
>
> if (Math.abs(a - b) < diff) {
>     System.out.println("true");
> }
> ```
>
> Positive example (2) — BigDecimal (construct from `String`, never from `double`):
> ```java
> BigDecimal a = new BigDecimal("1.0");
> BigDecimal b = new BigDecimal("0.9");
> BigDecimal c = new BigDecimal("0.8");
>
> BigDecimal x = a.subtract(b);
> BigDecimal y = b.subtract(c);
>
> if (x.equals(y)) {
>     System.out.println("true");
> }
> ```

**9. [Mandatory]** Rules for using primitive data types and wrapper classes:
1. Members of a POJO class **must** be wrapper classes.
2. The return value and arguments of an RPC method **must** be wrapper classes.
3. [Recommended] Local variables should be primitive data types.

> Rationale: `null` carries meaning in DTOs and over RPC ("unknown" / "not provided") that a primitive cannot express. A primitive `int` field always reads `0` — indistinguishable from "the caller set it to zero".

**10. [Mandatory]** While defining POJO classes like DO, DTO, VO, etc., do not assign any default values to the members.

**11. [Mandatory]** To avoid a deserialization failure, do not change the `serialVersionUID` when a serialized class needs to be updated.

> Note: Inconsistency of `serialVersionUID` may cause an `InvalidClassException` at runtime. If the class structure changes incompatibly, change the UID intentionally.

**12. [Mandatory]** Business logic in constructor methods is prohibited. All initializations should be implemented in the `init` method.

**13. [Mandatory]** The `toString` method must be implemented in a POJO class. The `super.toString` method should be called in the beginning if the current class extends another POJO class.

**14. [Recommended]** When accessing an array generated by the `split` method in `String` using an index, make sure to check the last separator whether it is null.

> Counter example:
> ```java
> String str = "a,b,c,,";
> String[] ary = str.split(",");
> // The expected result is 5 but it turns out to be 3.
> System.out.println(ary.length);
> ```

**15. [Recommended]** Multiple constructor methods or homonymous methods in a class should be put together for better readability.

**16. [Recommended]** The order of methods declared within a class is: `public` or `protected` methods → `private` methods → `getter`/`setter` methods.

**17. [Recommended]** For a `setter` method, the argument name should be the same as the field name. Implementations of business logic in `getter`/`setter` methods are not recommended.

> Counter example:
> ```java
> public Integer getData() {
>     if (true) {
>         return data + 100;
>     } else {
>         return data - 100;
>     }
> }
> ```

**18. [Recommended]** Use the `append` method in `StringBuilder` inside a loop body when concatenating multiple strings.

> Counter example:
> ```java
> String str = "start";
> for (int i = 0; i < 100; i++) {
>     str = str + "hello";  // each iteration allocates a new char[]
> }
> ```

**19. [Recommended]** Keyword `final` should be used in the following situations:
1. A class which is not allowed to be inherited, or a local variable not to be reassigned.
2. An argument which is not allowed to be modified.
3. A method which is not allowed to be overridden.

**20. [Recommended]** Be cautious to copy an object using the `clone` method in `Object`.

> Note: The default implementation of `clone` in `Object` is a shallow (not deep) copy.

**21. [Recommended]** Define the access level of members in a class with severe restrictions:
1. Constructor methods must be `private` if allocation using `new` is forbidden.
2. Constructor methods are not allowed to be `public` or default in a utility class.
3. Non-static class variables accessed from inheritors must be `protected`.
4. Non-static class variables no one can access except the class must be `private`.
5. Static variables no one can access except the class must be `private`.
6. Static variables should be considered for being `final`.
7. Class methods no one can access except the class must be `private`.
8. Class methods accessed from inheritors must be `protected`.

## Collection

**1. [Mandatory]** The usage of `hashCode` and `equals` should follow:
1. Override `hashCode` if `equals` is overridden.
2. These two methods must be overridden for elements of a `Set`.
3. These two methods must be overridden for any object used as the key of `Map`.

> Note: `String` can be used as the key of `Map` since `String` defines these two methods.

**2. [Mandatory]** Do not add elements to collection objects returned by `keySet()` / `values()` / `entrySet()`; otherwise `UnsupportedOperationException` will be thrown.

**3. [Mandatory]** Do not add nor remove to/from immutable objects returned by methods in `Collections`, e.g. `emptyList()` / `singletonList()`.

> Counter example: Adding elements to `Collections.emptyList()` will throw `UnsupportedOperationException`.

**4. [Mandatory]** Do not cast `subList` in class `ArrayList`; otherwise `ClassCastException` will be thrown — `java.util.RandomAccessSubList` cannot be cast to `java.util.ArrayList`.

> Note: `subList` of `ArrayList` is an inner class, which is a view of `ArrayList`.

**5. [Mandatory]** When using `subList`, be careful when modifying the size of the original list. It might cause `ConcurrentModificationException`.

**6. [Mandatory]** Use `toArray(T[] array)` to convert a list to an array. The input array type should be the same as the list.

> Counter example: Do not use `toArray()` without arguments — `ClassCastException` will be thrown.
>
> Positive example:
> ```java
> List<String> list = new ArrayList<String>(2);
> list.add("guan");
> list.add("bao");
> String[] array = new String[list.size()];
> array = list.toArray(array);
> ```

**7. [Mandatory]** Do not use methods which will modify the list after using `Arrays.asList` to convert array to list.

> Note: The result of `asList` is the inner class of `Arrays`, which does not implement methods to modify itself.
>
> ```java
> String[] str = new String[] { "a", "b" };
> List<String> list = Arrays.asList(str);
> // list.add("c");  // UnsupportedOperationException
> // str[0] = "X";   // mutating the array also changes the list (live view)
> ```

**8. [Mandatory]** Method `add` cannot be used for a generic wildcard with `<? extends T>`; method `get` cannot be used with `<? super T>`.

> Note (PECS — Producer Extends Consumer Super):
> - `extends` is suitable for frequently reading scenarios.
> - `super` is suitable for frequently inserting scenarios.

**9. [Mandatory]** Do not remove or add elements to a collection in a `for-each` loop. Please use `Iterator`.

> Counter example:
> ```java
> List<String> a = new ArrayList<String>();
> a.add("1");
> a.add("2");
> for (String temp : a) {
>     if ("1".equals(temp)) {
>         a.remove(temp);   // ConcurrentModificationException
>     }
> }
> ```
>
> Positive example:
> ```java
> Iterator<String> it = a.iterator();
> while (it.hasNext()) {
>     String temp = it.next();
>     if (deleteCondition) {
>         it.remove();
>     }
> }
> ```

**10. [Mandatory]** In JDK 7 and above, `Comparator` should meet three requirements, otherwise `Arrays.sort` and `Collections.sort` will throw `IllegalArgumentException`:
1. Comparing `x,y` and `y,x` should return opposite results.
2. If `x>y` and `y>z`, then `x>z`.
3. If `x=y`, then comparing `x` with `z` and `y` with `z` should return the same result.

> Counter example (violates rule 1 — never returns 0):
> ```java
> new Comparator<Student>() {
>     @Override
>     public int compare(Student o1, Student o2) {
>         return o1.getId() > o2.getId() ? 1 : -1;
>     }
> }
> ```

**11. [Recommended]** Set a size when initializing a collection if possible.

> Note: Better to use `new ArrayList<>(initialCapacity)` to initialize `ArrayList`. For `HashMap`, use `new HashMap<>(expectedSize / 0.75f + 1)`.

**12. [Recommended]** Use `entrySet` instead of `keySet` to traverse KV maps.

> Note: `keySet` iterates through the map twice (once to get keys, then once per `get(key)`). `entrySet` iterates only once and is more efficient. In JDK 8+, also consider `map.forEach((k, v) -> ...)`.

**13. [Recommended]** Carefully check whether a k/v collection can store `null` values.

| Collection | Key | Value | Super | Note |
|---|---|---|---|---|
| `Hashtable` | Null not allowed | Null not allowed | `Dictionary` | Thread-safe |
| `ConcurrentHashMap` | Null not allowed | Null not allowed | `AbstractMap` | Segment lock |
| `TreeMap` | Null not allowed | Null allowed | `AbstractMap` | Thread-unsafe |
| `HashMap` | Null allowed | Null allowed | `AbstractMap` | Thread-unsafe |

> Counter example: Confused by `HashMap`, many people think `null` is allowed in `ConcurrentHashMap`. Actually, `NullPointerException` will be thrown.

**14. [For Reference]** Properly use sort and order of a collection to avoid negative influence.

> Note: *Sorted* means iteration follows a specific sorting rule. *Ordered* means the order of elements in each traversal is stable.

**15. [For Reference]** Deduplication operations can be performed quickly since `Set` stores unique values only. Avoid using the `contains` method of `List` for that purpose.
